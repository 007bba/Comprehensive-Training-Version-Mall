# 商城项目解读与快速复现指南

这份文档的目标不是逐行解释所有代码，而是帮助你在较短时间内掌握这个商城项目的整体思路、功能实现方式和复现路线。项目实际包含两个主要部分：`myshop-vue` 是 Vue 前端，`myshop-api2` 是 Django REST Framework 后端。前端负责页面展示、路由跳转、用户交互和调用接口；后端负责商品、分类、购物车、订单、用户和地址等业务接口。

## 1. 项目整体定位

这是一个典型的前后端分离商城项目。用户在前端页面中浏览首页、查看商品分类、进入商品详情、加入购物车、登录注册、管理地址、提交订单、查看订单；前端通过 Axios 请求后端接口，后端通过 Django REST Framework 提供 REST API，并使用 JWT 处理登录后的接口认证。

项目最值得学习的地方有四条主线：第一是 Vue 单页应用的页面组织方式；第二是 Axios 封装接口请求的方式；第三是 Vuex 如何保存登录用户和购物车状态；第四是 Django REST Framework 如何用模型、序列化器、视图集和路由快速搭建商城接口。

## 2. 技术栈概览

前端位于 `myshop-vue`，主要技术是 Vue 2、Vue Router、Vuex、Axios、Webpack 3、vue-awesome-swiper。入口文件是 `src/main.js`，根组件是 `src/App.vue`，路由配置是 `src/router/index.js`，全局状态在 `src/vuex/store.js`，接口封装集中在 `src/api` 和 `src/utils/request.js`。

后端位于 `myshop-api2`，主要技术是 Django 3.1.5、Django REST Framework、djangorestframework-jwt、django-filter、django-cors-headers、drf-extensions、django-ckeditor、MySQL。入口管理文件是 `manage.py`，项目配置在 `myshop/settings.py`，总路由在 `myshop/urls.py`，业务应用分为 `apps/goods`、`apps/order`、`apps/users`、`apps/basic`。

## 4. 前端整体运行思路

前端从 `src/main.js` 启动。它创建 Vue 实例，把 `router` 和 `store` 注入进去，并挂载到 `#app`。这里还配置了一个全局路由守卫：每次路由变化时，如果目标路由设置了 `meta.title`，就把页面标题改成对应标题。

`src/App.vue` 非常简单，只放了一个 `<router-view/>`。所以真正的页面由 `src/router/index.js` 决定。路由中定义了首页、商品列表、商品详情、购物车、注册、登录、结算、收货地址、个人中心、我的订单等页面。

前端页面的共同模式是：页面组件在 `created()` 生命周期中调用接口方法；接口方法从 `src/api` 引入；接口方法最终调用 `src/utils/request.js` 里的 Axios 实例；请求成功后把返回的 `response.data.data` 放到当前组件的 `data` 中渲染。

## 5. 后端整体运行思路

后端是 Django REST Framework 项目。数据库模型定义业务实体，序列化器决定接口输入输出格式，视图或视图集处理请求，路由把 URL 映射到视图。

后端总路由 `myshop/urls.py` 把四个业务应用挂到根路径下，所以前端访问的 `/goods/`、`/cart/`、`/users/`、`/address/` 等接口都直接位于后端根路径。登录接口是 `/login/`，使用 `rest_framework_jwt.views.obtain_jwt_token`，登录成功后返回 JWT token 和用户信息。

后端自定义了 `CustomModelViewSet` 和 `CustomListModelMixin` 等通用类，把 DRF 默认响应包装成统一格式，例如：

```json
{
  "code": 200,
  "msg": "OK",
  "data": []
}
```

商品分页也使用自定义响应，额外返回 `count`、`next`、`previous` 等分页信息。

## 7. 登录认证流程

登录页面在 `views/user/login.vue`。用户输入用户名和密码后调用 `src/api/index.js` 中的 `login()`，实际请求 `POST /login/`。后端使用 JWT 登录视图，认证逻辑经过 `apps/users/views.py` 中的 `CustomBackend`。这个自定义认证后端允许用户用用户名或手机号登录。

登录成功后，前端做了两件事：第一，把 `Response.data.token` 存入 `localStorage` 的 `token`；第二，通过 `this.$store.dispatch("saveUser", Response.data)` 把用户信息保存到 Vuex。之后页面跳转到 `/index`。

后续访问购物车、订单、地址、用户资料等需要登录的接口时，`src/utils/request.js` 的请求拦截器会判断 URL。如果 URL 是 `/cart/`、`/order/`、`/checkout/`、`/myorder/`，或者包含 `address/`、`users/` 等，就从 `localStorage` 取出 token，并加到请求头：

```text
Authorization: JWT <token>
```

如果接口返回 401，响应拦截器会弹出“请先登录！”。

## 8. 商品模块实现思路

商品模块分三层：分类、商品、首页轮播。

后端模型在 `apps/goods/models.py`。`GoodsCategory` 是分类表，支持父子分类，字段 `parent` 指向自己，所以可以形成一级分类和二级分类。`Goods` 是商品表，保存商品名称、分类、市场价、销售价、销量、库存、详情、主图、是否推荐等信息。`Slide` 是首页轮播图，关联一个商品和一张轮播图片。

后端接口在 `apps/goods/views.py`。`GoodsCategoryViewset` 返回顶级分类，并通过序列化器带出子分类。`IndexCategoryGoodsViewSet` 返回首页分类楼层数据，每个顶级分类下面会查出它自己和子分类下的商品。`GoodsView` 返回商品列表和详情，支持分页、搜索、排序、价格过滤、分类过滤。详情接口重写了 `retrieve()`，每访问一次商品详情就让 `click_num` 加 1。

前端首页由 `views/index/index.vue` 组合了四个组件：公共头部、轮播、推荐商品、分类商品、底部。头部 `views/app/head.vue` 会请求分类接口生成导航菜单；轮播 `views/index/slide.vue` 请求 `/slide/`；推荐商品 `views/index/recommend.vue` 请求 `/goods/`；分类楼层 `views/index/categorygoods.vue` 请求 `/indexgoods/`。

商品列表页 `views/list/list.vue` 接收路由参数 `/list/:id` 中的分类 id，然后请求分类详情和商品列表。商品列表请求会带上分类、最低价、最高价、排序字段和页码。分页组件 `views/list/page.vue` 不直接请求后端，而是通过 `$emit("get_page")` 通知父组件重新请求数据。

商品详情页 `views/detail/detail.vue` 接收 `/detail/:id`，调用 `getGoodsByID(id)` 获取商品信息。页面上可以增加或减少购买数量，点击加入购物车时调用 `addCart({ goods: this.goods_id, goods_num: this.buynum })`。

## 9. 购物车模块实现思路

购物车前端涉及三个地方：商品详情页负责加入购物车，购物车页负责展示和修改数量，公共头部负责显示购物车商品数量和总价。

核心状态在 `src/vuex/store.js`。`state.cart_lists` 保存购物车列表；`saveCart` mutation 会调用后端 `getCart()`，把购物车数据保存到 `state.cart_lists` 和 `localStorage`；`totalNum` getter 根据每项 `goods_num` 计算总数量；`totalPrice` getter 根据 `item.goods.price * item.goods_num` 计算总价。

购物车后端模型是 `apps/order/models.py` 中的 `Cart`。它关联用户和商品，并保存购买数量。后端视图是 `CartViewset`。这个视图集要求用户登录，只返回当前用户自己的购物车数据。

加入购物车的关键逻辑在 `CartModelSerializer.create()`：如果当前用户购物车中已经有这个商品，就不新建记录，而是在原记录上累加数量；如果没有，就创建新记录。购物车详情序列化器 `CartDetailModelSerializer` 会把商品完整信息嵌套返回给前端，所以前端可以直接显示 `item.goods.name`、`item.goods.price`、`item.goods.main_img`。

购物车数量修改通过 `PATCH /cart/<goods_id>/` 完成。后端 `CartViewset` 设置了 `lookup_field = "goods_id"`，所以 URL 中传的是商品 id，而不是购物车记录 id。`perform_update()` 会根据数量变化同步调整商品库存。

需要注意一点：当前代码在“加入购物车”和“提交订单”两个阶段都有库存减少逻辑。`CartViewset.perform_create()` 会减少库存，`OrderView.post()` 里提交订单时也会减少库存。这种设计在真实项目中容易导致库存被扣两次。作为学习项目，你可以先理解它的意图：加入购物车和修改购物车时维护库存；提交订单时再次校验库存并生成订单。复现时建议把库存扣减统一放到“提交订单”阶段，更符合常见业务设计。

## 10. 订单模块实现思路

订单前端主要是 `views/cart/checkout.vue` 和 `views/user/myorder.vue`。

结算页从 Vuex 读取购物车列表和总价，从地址接口读取用户地址，并默认取地址列表第一条作为收货信息。用户点击提交订单时，请求 `POST /order/`，提交联系人、手机号、地址、支付方式和订单备注。提交成功后，前端调用 `this.$store.dispatch("saveCart")` 重新刷新购物车，并跳转到我的订单页面。

后端订单模型有两张表：`Order` 是订单主表，保存订单号、总数量、总金额、收货地址、联系人、手机号、支付方式、备注、订单状态、用户和创建时间；`OrderGoods` 是订单商品表，保存订单和商品的关系、购买数量以及下单时的价格快照。

订单创建逻辑在 `apps/order/views.py` 的 `OrderView.post()`。它做了以下事情：生成订单号；读取前端提交的收货和支付信息；开启事务保存点；创建订单主表；查询当前用户购物车；对每个购物车商品使用 `select_for_update()` 做数据库锁；判断库存是否足够；扣减库存、增加销量；创建订单商品记录；累计订单总数量和总金额；更新订单主表；删除购物车；提交事务；返回“订单生成成功”。

这是后端最值得重点学习的业务流程，因为它体现了订单类接口的典型写法：事务、库存校验、主从表写入、价格快照、清空购物车。

## 11. 用户和地址模块实现思路

用户模型是 `apps/users/models.py` 的 `MyUser`，继承 Django 的 `AbstractUser`，扩展了真实姓名、手机号、性别、生日、头像、会员等级、状态、创建时间和更新时间。

注册逻辑由 `MyUserRegSerializer` 负责。它校验用户名是否重复、手机号是否重复、手机号格式是否合法，并在 `create()` 中调用 `set_password()`，确保密码不是明文入库。

登录逻辑使用 JWT。项目在 `myshop/settings.py` 里配置了 `AUTHENTICATION_BACKENDS = ('apps.users.views.CustomBackend',)`，所以登录时会走 `CustomBackend.authenticate()`。它用 `Q(username=username) | Q(mobile=username)` 查询用户，因此用户名和手机号都可以登录。

个人资料页 `views/user/profile.vue` 调用用户接口获取和修改当前用户资料。虽然前端函数名写的是 `getUsersByID()`，URL 也写成 `/users/1/`，但后端 `MyUserViewSet.get_object()` 实际返回的是 `self.request.user`，所以它依赖当前登录用户，而不是固定的一号用户。

地址模型是 `apps/basic/models.py` 的 `Address`。地址接口 `AddressViewset` 要求登录，只返回当前用户的地址。序列化器中 `user` 使用 `HiddenField(default=CurrentUserDefault())`，所以前端新增地址时不需要传 user，后端会自动绑定当前登录用户。

地址序列化器里还有一个默认地址逻辑：如果新增或修改的地址 `is_default == 1`，就把其他默认地址改为 0。当前实现是全表更新，没有按当前用户过滤；真实项目中应该只更新当前用户自己的地址。

## 12. Vuex 在项目中的作用

`src/vuex/store.js` 是全局状态中心。它保存三个主要状态：`count` 示例计数器、`userinfo` 登录用户信息、`cart_lists` 购物车列表。

最重要的是用户和购物车。用户登录后调用 `saveUser`，退出登录调用 `delUser`。购物车发生变化后调用 `saveCart`，由 Vuex 统一重新请求后端购物车列表。头部组件通过 `mapGetters` 读取 `userinfo`、`cart_lists`、`totalNum`、`totalPrice`，所以头部能实时显示登录用户和购物车汇总。

不过当前 `saveUser` 中有一个值得注意的小问题：`localStorage.setItem('userinfo', value)` 直接保存对象时会变成字符串 `[object Object]`；getter 里又会 `JSON.stringify(state.userinfo)`。如果你复现项目，建议统一使用 `JSON.stringify` 保存，用 `JSON.parse` 读取。

## 13. Axios 封装和接口认证

前端真正使用较多的是 `src/utils/request.js`。它创建了一个 Axios 实例：

```js
const service = axios.create({
  baseURL: 'http://localhost:8000/',
  timeout: 6000
})
```

请求拦截器负责判断接口是否需要 JWT。如果需要，就给请求头加上 `Authorization`。响应拦截器负责统一处理 401。

`src/axios/index.js` 是另一套较早的 Axios 封装，但主业务代码基本走 `src/utils/request.js`。学习时可以重点看 `utils/request.js`，不必把两套封装都当成主线。

## 14. 数据模型关系

可以把后端数据关系理解成下面这样：

```text
MyUser 用户
  ├── Address 收货地址
  ├── Cart 购物车项
  │     └── Goods 商品
  └── Order 订单主表
        └── OrderGoods 订单商品明细
              └── Goods 商品

GoodsCategory 商品分类
  ├── parent 自关联形成父子分类
  └── Goods 商品

Slide 首页轮播
  └── Goods 商品
```

商城业务的本质就是围绕这几类数据流转：分类决定商品怎么展示；商品进入购物车；购物车生成订单；订单绑定用户、地址和商品明细。

## 15. 主要页面流程

首页流程：进入 `/index`，加载头部分类菜单、轮播图、推荐商品、分类楼层商品。用户可以从导航或商品卡片进入列表页或详情页。

商品列表流程：进入 `/list/:id`，根据分类 id 获取当前分类和子分类，再请求商品列表。用户可以按价格区间筛选、按销量或价格排序、点击分页切换页面。

商品详情流程：进入 `/detail/:id`，请求商品详情。用户选择数量后加入购物车。加入成功后弹窗提示，可以继续购物或去购物车。

购物车流程：进入 `/cart`，请求当前登录用户购物车。用户可以增加或减少商品数量，页面重新请求购物车并刷新 Vuex 中的购物车数据。

结算流程：进入 `/checkout`，读取 Vuex 中的购物车商品和总价，请求地址列表，展示订单确认页。提交订单后后端创建订单、订单明细并删除购物车。

用户流程：注册页提交用户名、密码、手机号；登录页提交用户名或手机号和密码；登录成功后保存 token 和用户信息；个人中心可以修改资料、管理地址、查看订单。

## 16. 如果你要快速掌握，建议按这个顺序学

第一步，先跑通项目。前端默认访问后端 `http://localhost:8000/`，前端开发服务默认是 `localhost:8080`。先确保后端能启动，前端能访问首页。

第二步，只看主流程，不纠结样式。你先从 `src/router/index.js` 记住页面有哪些，再从 `/index -> /list/:id -> /detail/:id -> /cart -> /checkout -> /myorder` 串一遍。商城项目的主线就是“浏览商品、加入购物车、提交订单”。

第三步，对照接口看页面。打开一个页面时，先找它 `created()` 里调用了哪个方法，再找这个方法从哪个 `api/*.js` 文件引入，再看后端哪个 URL 和视图处理它。

第四步，重点掌握四个后端模块：商品、用户、购物车、订单。商品模块学习筛选、排序、分页和详情点击数；用户模块学习自定义用户、注册校验和 JWT 登录；购物车模块学习当前用户数据隔离和重复商品累加；订单模块学习事务、库存、订单主表和明细表。

第五步，自己复现一个最小版本。不要一开始复刻所有 CSS 和页面细节，只复现核心功能：登录注册、商品列表、商品详情、加入购物车、购物车列表、提交订单、我的订单。样式可以最后补。

## 17. 推荐的最小复现路线

如果时间赶，建议按“后端接口优先、前端页面跟进”的方式复现。

第一阶段，复现后端数据模型：创建用户、分类、商品、购物车、订单、订单商品、地址模型。先让 Django admin 能添加分类和商品。

第二阶段，复现后端接口：实现 `/login/`、`/users/`、`/goodscategory/`、`/goods/`、`/goods/<id>/`、`/cart/`、`/order/`、`/address/`。接口返回格式可以先简单，跑通后再统一封装响应。

第三阶段，复现前端基础框架：创建 Vue 2 项目，配置 Vue Router、Vuex、Axios。先做路由页面骨架，不急着写完整样式。

第四阶段，复现商品浏览：做首页、分类列表、商品详情。能从首页进入列表，从列表进入详情即可。

第五阶段，复现登录和认证：登录成功保存 token，Axios 请求拦截器自动带 token。确认购物车接口未登录时会报 401。

第六阶段，复现购物车和订单：详情页加入购物车，购物车页查看和修改数量，结算页提交订单，订单页展示订单列表。

第七阶段，补齐地址和个人中心：实现地址增改查，结算页默认取一个地址，个人中心能查看和修改用户资料。

## 18. 复现时必须抓住的几个关键点

第一个关键点是路由。前端每个页面都是由 `src/router/index.js` 映射出来的，复现时先把页面路径搭好，后面功能才好串。

第二个关键点是接口封装。不要在每个页面里直接写 Axios 请求，应该像原项目一样把接口放到 `src/api`，页面只调用业务方法。

第三个关键点是 token。登录后保存 token，请求需要登录的接口时自动带上 token，这是用户、购物车、订单、地址模块能串起来的前提。

第四个关键点是 Vuex。购物车总数和总价适合放到 getter 里算，这样头部、购物车页、结算页都能共享。

第五个关键点是订单事务。订单不是简单插一条记录，而是从购物车读取商品，校验库存，创建订单主表和明细表，更新库存和销量，删除购物车。这是商城项目后端最核心的业务能力。

## 19. 当前项目中可以改进或要留意的地方

这个项目适合学习，但代码里有一些学习项目常见的问题。你复现时可以先照着理解，真正重写时建议优化。

第一，前端 `api/index.js`、`api/goods.js`、`api/users.js` 中有重复的登录、注册等接口封装。复现时建议按模块统一放置，比如登录注册都放 `api/users.js`。

第二，前端存在多个备份文件，例如 `checkout copy.vue`、`checkout - 副本.vue`、`store copy.js`。学习主线时不要看这些备份文件，以正式命名文件为准。

第三，`src/utils/request.js` 里判断是否需要 token 的方式是硬编码 URL。复现时可以简单保留这种方式；如果想更规范，可以让所有需要认证的 API 都自动带 token，或者通过接口配置控制。

第四，用户资料接口前端写死 `/users/1/`，但后端实际返回当前登录用户。这个写法容易误导，复现时建议改成 `/users/me/` 这种更清楚的接口，或者至少前端函数不要叫 `getUsersByID`。

第五，购物车和订单都有库存扣减逻辑，可能造成重复扣库存。更推荐的设计是购物车只保存意向购买数量，不扣库存；提交订单时再校验并扣库存。

第六，地址默认值更新没有按当前用户过滤，可能影响其他用户。复现时应写成只更新当前用户的其他地址。

第七，后端配置文件里包含本地数据库账号等敏感配置。学习时知道它需要 MySQL 连接即可，真正项目中应该用环境变量或本地配置文件，不要把敏感信息写进代码仓库。

第八，前端部分链接还保留了旧式 `href="flow.php"`、`href="goods.php"`、外部注册地址等痕迹。这些更像从传统商城模板改造过来的页面，复现时应统一改为 Vue Router 路由。

## 20. 你应该重点背下来的项目思路

这个商城项目的核心思路可以浓缩成一句话：前端用 Vue Router 组织页面，用 Axios 调用后端接口，用 Vuex 保存登录和购物车状态；后端用 Django REST Framework 按业务模块建立模型、序列化器、视图集和路由，通过 JWT 识别当前用户，并围绕商品、购物车、订单完成商城业务闭环。

如果你要向别人讲这个项目，可以按这条线讲：用户进入首页后，前端调用商品分类、轮播和商品接口展示数据；用户登录后拿到 JWT，之后购物车、地址、订单接口都会带上 token；用户在详情页加入购物车，后端按当前用户保存购物车；结算时后端从当前用户购物车生成订单主表和订单商品表，并更新库存、销量、清空购物车；用户最后在我的订单页面查看订单。

## 21. 短时间学习安排建议

如果你只有很短时间，建议不要平均用力。先花主要精力掌握“商品浏览、登录认证、购物车、下单”这四条主链。公共头尾、CSS、轮播、分页组件可以放后面。

第一轮只看功能流：打开路由文件，知道每个 URL 到哪个页面；打开每个页面的 `created()`，知道页面请求哪个接口。

第二轮看数据流：从前端 API 文件跳到后端 URL，再跳到后端视图、序列化器和模型。重点弄懂 `goods -> cart -> order` 的数据变化。

第三轮自己画图或默写：默写出主要表关系、主要接口、主要页面流程。能不看代码说出“加入购物车”和“提交订单”的流程，就说明你真正掌握了。

第四轮动手复现最小版：不要复制样式，先做一个功能能跑通的版本。能跑通后再回头对照原项目补页面细节。

## 22. 面试或答辩时可以这样讲

这个项目是一个前后端分离商城系统。前端采用 Vue 2、Vue Router、Vuex 和 Axios，后端采用 Django REST Framework 和 MySQL。前端负责页面展示和用户交互，后端提供 REST API。用户登录后后端返回 JWT，前端保存 token，并在访问购物车、订单、地址等接口时自动把 token 放入请求头。商品模块支持分类、列表、详情、价格筛选、排序和分页；购物车模块按当前登录用户保存商品和数量；订单模块从购物车生成订单主表和订单商品明细，并处理库存、销量和清空购物车。项目的核心业务闭环是：商品展示、加入购物车、确认订单、生成订单、查看订单。

## 23. 最后建议

你现在最需要的不是把每个 CSS 和模板细节都背下来，而是抓住项目主线并能独立复现。建议你把这个文档作为地图，然后按“路由 -> 页面 -> API -> 后端视图 -> 序列化器 -> 模型”的顺序逐个功能过一遍。每过一个功能，就用自己的话写一句流程说明。等你能完整讲出登录、商品列表、商品详情、购物车、下单这五个流程时，这个项目就基本掌握了。
